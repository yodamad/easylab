Overrides the wildcard record derived from the domain above. Leave blank unless you need a different base — with a DNS provider configured, the wildcard record is created automatically.

DNS will not be managed for you

Workspace URLs are {workspace}.{domain}, so you must create a wildcard A record for *.your-domain pointing at the lab's ingress IP once it is provisioned — otherwise workspace URLs will not resolve and their certificates stay pending.

Each workspace will also request its own Let's Encrypt certificate, capped at 50 per registered domain per week. Selecting a DNS provider below issues a single wildcard certificate instead.

DNS Provider (Optional)

Automates DNS A-record creation and enables wildcard TLS certificates via DNS-01 challenge.

The DNS zone that contains the domain above.
OVH application must have /domain/zone/* permissions. Enables wildcard TLS and automated A-record creation.
Service principal must have DNS Zone Contributor role on the DNS zone resource group.
Installs ExternalDNS in the cluster to create and delete one DNS record per workspace, rather than a single wildcard record covering all of them. Use this when your DNS administrator will not grant a wildcard. Records appear about a minute after a workspace starts; the student portal waits for the hostname to resolve before showing it as ready.

Build the workspace from the workshop repository's .devcontainer/devcontainer.json. The image, Dockerfile and features are built in each student's workspace on first start. How students see this workspace. Lowercased, with dashes for anything else. Must be unique within the lab — the devcontainer's own name is a display string that is often the same across repos.
The repository field above is still used for the generated template's git_repo, since that is what students clone.
Required. Without it every student rebuilds the devcontainer from scratch.
code-server is injected into the image the devcontainer builds, so it is independent of the devcontainer's own base image. The devcontainer must be built on a glibc base — the injected IDE cannot run on Alpine/musl.
If the workshop repository is private, the git credential (from the Credentials section above) used to read the devcontainer during import and, baked into the generated template, to clone the repo in each student's workspace. With a single git credential this is applied automatically.
If the devcontainer builds from a private base image or pushes to a private cache registry, the registry credential (from the Credentials section) each student's workspace pulls and pushes it with. With a single registry credential this is applied automatically.
